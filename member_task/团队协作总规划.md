### 1.2 为什么这样分权重？

| 成员 | 工作量依据 |
|------|-----------|
| **成员1** | 所有成员的后端代码都依赖他的脚手架和数据库，是阻塞性工作，第1~3天必须交付 |
| **成员3** | 商品模块是平台核心价值，后端6个API + 前端5个页面 + 搜索模块，代码量最大 |
| **成员4** | 订单状态机设计复杂度高，涉及买卖双方交互逻辑，还需维护收藏模块 |
| **成员5** | 前端工程搭建阻塞所有前端开发，公共组件被全员复用，私信系统需额外设计 |
| **成员2** | 用户模块API数量多（9个）但业务逻辑相对标准，前端页面较常规 |
| **成员6** | 管理后台MVP阶段仅需审核+基础统计，功能最少，适合能力稍弱的组员或用时较少者 |
| **组长** | 不直接写业务代码，负责架构设计、代码审查、进度管理、集成测试、风险把控 |

## 二、MVP范围定义

### 2.1 ✅ 本次2周要做（核心闭环）

| 模块 | 范围 | 说明 |
|------|------|------|
| 用户模块 | 注册(手机号+固定验证码)、登录(JWT)、个人信息、地址管理 | 学号用正则校验，无短信服务 |
| 商品模块 | 发布、编辑、删除、上下架、列表浏览、详情页、类目树 | 含图片上传（本地存储） |
| 搜索模块 | 关键词搜索(MySQL LIKE)、按分类/价格筛选 | 不做Elasticsearch |
| 订单模块 | 下单、状态流转(待沟通→待自提→已完成)、订单查询 | 状态机逻辑 |
| 私信模块 | HTTP版消息发送/接收、会话列表 | 不做WebSocket，降级为HTTP轮询或简单请求 |
| 收藏模块 | 收藏/取消收藏、收藏列表 | toggle模式 |
| 管理模块 | 商品审核(通过/驳回)、基础数据统计 | 仅1个管理员角色，无图表 |
| 文件模块 | 图片上传（本地磁盘存储） | 不做缩略图，不做MinIO |

### 2.2 ❌ 本次不做（后续迭代）

| 砍掉的功能 | 理由 |
|------------|------|
| 求购模块（发布/智能匹配） | Phase 2，非核心闭环 |
| WebSocket实时聊天 | HTTP版消息已满足MVP通信需求 |
| 短信验证码服务 | 用固定验证码"888888"替代 |
| 管理员三级权限体系 | 简化为单一管理员角色 |
| 数据看板图表（ECharts） | 仅展示数字统计 |
| Docker Compose部署 | 本地直接运行Spring Boot + MySQL |
| Elasticsearch搜索 | MySQL LIKE足够校园数据量 |
| 图片缩略图生成 | 前端做CSS缩放 |
| 敏感词过滤 | 后期加入 |
| 接口限流 | 本地演示不需要 |
| 评价信用体系 | Phase 2 |
| 系统化预约自提 | 订单备注里沟通即可 |

## 五、协作规范

### 5.1 代码管理

```
Git分支策略（简化版）：
  main                ← 稳定分支，每日合并已验证代码
    ├── member-1/*    ← 成员1的功能分支
    ├── member-2/*    ← 成员2的功能分支
    ├── member-3/*    ← 成员3的功能分支
    ├── member-4/*    ← 成员4的功能分支
    ├── member-5/*    ← 成员5的功能分支
    └── member-6/*    ← 成员6的功能分支

命令规范：
  提交信息：[模块] 简短描述
  例：[user] 实现手机号注册API
  例：[product] 完成商品列表分页查询
```

### 5.2 每日站会（15分钟）

```
时间：每天 09:30
地点：线下/腾讯会议

议程：
1. 昨天完成了什么？（每人30秒）
2. 今天计划做什么？（每人30秒）
3. 有什么阻塞？需要谁协助？
4. 接口/文档更新通知

组长记录阻塞项并协调解决
```

### 5.3 接口文档规范

- 后端API写完**立即**更新Knife4j/Swagger注解
- API接口变更**第一时间**在群里通知相关前端成员
- 所有接口统一返回 `Result<T>` 格式

### 5.4 代码提交规范

- 每天至少提交1次代码（下班前）
- 提交前确保代码可编译通过
- 不要直接在main分支上开发
- 合并前组长做快速Code Review

## 六、风险与应急预案

| 风险 | 概率 | 应对 |
|------|------|------|

## 七、验收标准（Day 10）

### 核心流程必须跑通：

```
1. 用户注册 → 登录 → 查看个人信息 ✓
2. 发布商品(含图片) → 在首页看到 → 点击查看详情 ✓
3. 搜索商品 → 按分类/价格筛选 → 找到目标商品 ✓
4. 收藏商品 → 在收藏列表查看 ✓
5. 发起私信 → 对方收到消息 ✓
6. 下单 → 订单状态流转(待沟通→待自提→已完成) ✓
7. 管理员登录 → 审核商品 → 查看基础统计 ✓
```

### 不要求：
- ❌ 不要求性能优化（100人并发等）
- ❌ 不要求生产级安全（HTTPS、防攻击等）
- ❌ 不要求完善的错误处理
- ❌ 不要求移动端完美适配
- ❌ 不要求部署上线
