**状态机核心逻辑**（用枚举+switch实现）：
```
待沟通(status=0) → 待自提(status=1)：双方确认 → 下一状态
待自提(status=1) → 已完成(status=2)：双方确认交易完成
待沟通/待自提    → 已取消(status=3)：任一方取消，需记录取消原因+取消方
❌ 不允许跨状态跳跃（如 待沟通→已完成）
❌ 已完成/已取消状态不可再变更
```

## 关键提醒

- **状态机是你最核心的设计**，一定要用枚举严格控制合法转换路径，写单元测试覆盖所有状态转换
- 订单号生成规则简化为时间戳+随机数，如：`20260708143256`
- 订单取消时要记录`cancel_reason`和`cancel_by`（buyer/seller），方便后期追溯
- 收藏toggle利用数据库唯一索引是最简洁的实现，不需要先查再判断
- 订单列表的分页查询记得处理空状态（没有订单时显示"暂无订单"）
